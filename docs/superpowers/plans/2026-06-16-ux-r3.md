# App UX Round 3 (Onboarding/Us) Implementation Plan

> Use the `superpowers:executing-plans` skill (or `subagent-driven-development` for parallel tasks) to execute this plan. Each task is independently committable; flagged **combine-to-typecheck** pairs must land together to keep `tsc` green (a hook/type change and its first consumer).

## Goal

Bring the love_place **onboarding / 우리(Us)** experience to "real-app" quality for spec §R3:

1. **Invite-code persistence** — a PENDING couple re-hydrates its active code on mount (idempotent `create_invite` re-call; cheapest, no schema change) and renders it with copy/share. Connect-screen polish: `alert()` → global toast, auto-extract a valid 8-char code from pasted share text, auto-submit/focus on a valid code, and split create-vs-accept errors.
2. **Onboarding steps ② + ③** — post-connect, on a **dedicated guarded route `/onboarding/steps`** that a **consent-aware `RequireAuth`** routes ACTIVE-but-not-consented users to (so BOTH inviter and accepter must complete it — not bypassable): pick **MY** color from a distinct palette (default contrasting: inviter=blue `#3b6db5` / accepter=pink `#c25d86`); record location/photo **mutual consent** (timestamp = consent, per security-privacy §3.2) backed by an idempotent migration. A **profile editor** (display_name + color) reachable from Us.
3. **Partner identity** — render partner `avatar_url` (today ignored) with initial fallback; allow a **local nickname** when display_name is blank; warm "함께한 지 D+N" since `connected_at`.
4. **Disconnect** — honest copy (shared data becomes inaccessible after disconnect because `current_couple_id()` is ACTIVE-only), **export-before-disconnect required** (in-dialog download; 해제 disabled until export acknowledged).
5. **Real export** — photo ZIP + JSON (original blobs from Storage; **client-side `fflate`**, equal for both partners while ACTIVE).
6. **Auth resilience** — OAuth-first (kept); magic-link **resend cooldown timer** + friendlier rate-limit copy + **6-digit OTP fallback** (`verifyOtp`) for the mobile-Safari cross-context PKCE trap; robust callback (retry + same-browser guidance).
7. **Trash generalization** — surface restore for events/visits/photos/trips/itineraries (DB trash RLS already in 0010/0012) with deletion date + auto-purge horizon; **immediate Undo** at deletion (R1.5 toast pattern).
8. **Dialog primitive** — extract one `Dialog` (portal + backdrop + focus-trap + ESC + restore-focus); reuse for the disconnect confirm.
9. **Brand-new-user value preview** (spec R3 line 51 "브랜드뉴 사용자 가치 미리보기 or 솔로 모드") — a pre-connect "둘이 쓰면 이런 게 가능해요" value-preview block on `ConnectPage` so the connect screen isn't a dead gate for someone who hasn't paired yet (warm, action-oriented, not a tutorial). **Decision:** we ship the *value-preview* arm, NOT a *solo mode*. Rationale: every share table is `couple_id`-scoped with ACTIVE-only `current_couple_id()` RLS — a real solo (pre-pair) write path would require a parallel un-coupled data model + RLS, far beyond R3; the spec offers "or", so the tractable arm is the preview. (Full solo/standalone usage deferred to a later round if ever desired.)

## Architecture

- **DB**: one new idempotent migration `0014_consent_purge.sql` (0012-style: `ADD COLUMN IF NOT EXISTS` / `DROP POLICY IF EXISTS`+`CREATE`), applied via `npx supabase db push`:
  - Consent storage = **Option A** (dossier 02 §5 recommended): `profiles.location_consent_at timestamptz` + `photo_consent_at timestamptz` (self-owned, reuses existing `profiles_self_update` / `profiles_self_or_partner_select` — no new policy).
  - Auto-purge horizon: `deleted_at`-based, surfaced client-side (`deleted_at + 30d`); a `purge_trashed()` SECURITY DEFINER function for a future scheduled job (no pg_cron dependency required to land R3 — the **client surfaces** the horizon; the function is the purge mechanism for relationship-end/legal use).
  - **No invite-code read-policy change needed** — invite-code persistence uses the idempotent `create_invite()` re-call (dossier 01 §1 recovery option 1), so the read-back RLS question is sidestepped.
- **Color column** already exists (`profiles.color`, 0001) — no migration for color. Default-contrast is applied **client-side** by user_a/user_b role (no `accept_invite` change).
- **Export ZIP**: client-side `fflate` (added to `package.json` in the export task). Rationale below.
- **Pure logic** (DOM-free, vitest-first): `extractInviteCode()`, `daysTogether()`, `purgeDate()`, `defaultColorForRole()`, `partnerLabel()` in `src/lib/...`; `buildExportZip()` (fflate, takes injected blobs — testable without network).
- **Hooks**: `useUpdateProfile` (version-bumped self-update), `useConsent`/`useUpdateConsent`, generalized `useTrash`/`useRestore` (parameterized over table; `softDelete`/`restore` already table-agnostic), `useResendCooldown`, OTP-verify path on `useSignInWithOtp`.
- **Components**: new `src/components/common/Dialog.tsx` (+ css); `ProfileEditor`, `ColorPicker`, `ConsentStep`, `OnboardingSteps` wizard mounted at a **dedicated guarded route `/onboarding/steps`** (NOT inside `ConnectPage` — see routing decision below); generalized `TrashSection` (heterogeneous `kind` badge). Refactors: `ConnectPage`, `UsPage`, `LoginPage`, `AuthCallbackPage`, **`RequireAuth`** (consent-aware), **`router.tsx`** (new steps route).
- **Routing / consent gate (decision — fixes the reachability/race trap):** the §10.3 mutual-consent step is a CRITICAL legal item and must NOT be bypassable. The naïve "render `<OnboardingSteps/>` inside `ConnectPage` when ACTIVE-but-not-consented" is **unreachable**: `ConnectPage` mounts only at `/onboarding`, and `RequireAuth` currently does `if (active && onboarding) return <Navigate to="/" replace/>` — so the instant a couple becomes ACTIVE the guard bounces off `/onboarding` to `/` (fatal for the **inviter**, who becomes ACTIVE while not on the connect page at all; a redirect race for the accepter). **Fix = make the guard consent-aware + give steps ②③ their own guarded route:**
  - **Single source of truth for "consent recorded"** = `useConsent()` (Task 8) → `consentRecorded = location_consent_at != null && photo_consent_at != null` on the **caller's own** `profiles` row (self-owned, dossier 02 §5 Option A). Both inviter and accepter are gated on their OWN consent, so both must complete ②③.
  - **New route `/onboarding/steps`** (guarded, full-screen sibling of `/onboarding`) renders `<OnboardingSteps/>`.
  - **`RequireAuth` new logic** (Task 8a): not-ACTIVE → `/onboarding` (unchanged); ACTIVE & **not** consentRecorded → `/onboarding/steps`; ACTIVE & consentRecorded → `/` (only then). `ConnectPage` (`/onboarding`) just navigates to `/onboarding/steps` on ACTIVE; it no longer tries to host the wizard.

### Export approach (decision)

**Client-side `fflate`** (added as a dep), NOT an Edge Function. Why:
- Dossier 03 §6 recommends starting client-side for **parity/simplicity**; RLS already gives each member identical couple-scoped read while ACTIVE, so JSON+ZIP is **symmetric by construction** (equal for both partners) — no service-role re-implementation of couple-member auth needed.
- `fflate` (~8KB, streaming, TS types) is leaner than `jszip` (~100KB+) and is the dossier's explicit leaner pick. The prompt says "prefer client-side JSZip if simpler than an Edge Function — pick the tractable path and state why": client-side wins (no infra); `fflate` over `jszip` for size/streaming. If real albums later exceed browser memory, migrate to an `export-bundle` Edge Function (out of R3 scope).
- `photos.storage_url` contract (dossier 03 §2, OPEN): exporter treats it as an **object path inside the private `photos` bucket prefixed `<coupleId>/`** (matches blog-publish isolation) and downloads via `supabase.storage.from('photos').download(path)` (RLS-scoped). Degrades gracefully to **0 photos** today (no photo writer exists yet), so the ZIP always contains at least `data.json`. Column confirmed `storage_url text NOT NULL` (`0001_core_schema.sql:140`).
- **`photos` is NOT in `database.types.ts`** (hand-written stub defines only `regions/couples/profiles/places/wishes` — lines 14–132; no `photos/events/visits/trips/itineraries/reactions`). Consequences the plan accepts deliberately (do NOT try to "fix" by regenerating gen-types — Task 5 only extends `profiles`):
  - `fetchCoupleExport` already reads `photos` via the **untyped string-table path** (it loops the `EXPORT_TABLES` string-union with `supabase.from(t)`, not the typed client) — this works against the stub today and is unchanged by R3.
  - `supabase.storage.from('photos').download(path)` is storage (not table-typed) — no `Database` type involved.
  - Therefore the `PhotoRow` shape in Task 11 (`{ id: string; storage_url: string }`) and the cast in Task 12 (`data.tables.photos as { id: string; storage_url: string }[]`) are a **local hand-typed contract** validated against `0001` SQL, NOT derived from `Database`. This is acceptable under the stub; `supabase gen types` is out of R3 scope.
- Export originals are **NOT EXIF-stripped** (security-privacy §5.2 — private personal copies, distinct from public blog publish).

  ```tsx
  import { useEffect, useRef, type KeyboardEvent as ReactKeyboardEvent, type ReactNode, type RefObject } from 'react'
  import { createPortal } from 'react-dom'
  import styles from './Dialog.module.css'

  type DialogProps = {
    open: boolean
    onClose: () => void
    ariaLabel: string
    initialFocusRef?: RefObject<HTMLElement | null>
    className?: string
    children: ReactNode
  }

  const FOCUSABLE = 'button, input, textarea, select, [href], [tabindex]:not([tabindex="-1"])'

  // 공용 모달 프리미티브 — 포털 + 백드롭 + 포커스 트랩 + ESC + 복귀 포커스(EventSheet 패턴 일원화, §8 a11y).
  export function Dialog({ open, onClose, ariaLabel, initialFocusRef, className, children }: DialogProps) {
    const sheetRef = useRef<HTMLDivElement>(null)
    const prevFocusRef = useRef<HTMLElement | null>(null)

    // 열릴 때: 직전 포커스 저장 후 초기 포커스. 닫힐 때(언마운트): 직전 포커스 복귀.
    useEffect(() => {
      if (!open) return
      prevFocusRef.current = (document.activeElement as HTMLElement | null) ?? null
      const focusTarget =
        initialFocusRef?.current ??
        sheetRef.current?.querySelector<HTMLElement>(FOCUSABLE) ??
        sheetRef.current
      focusTarget?.focus()
      return () => {
        prevFocusRef.current?.focus()
      }
    }, [open, initialFocusRef])

    // ESC 닫기(window 리스너, cleanup).
    useEffect(() => {
      if (!open) return
      const onKey = (e: KeyboardEvent) => {
        if (e.key === 'Escape') onClose()
      }
      window.addEventListener('keydown', onKey)
      return () => window.removeEventListener('keydown', onKey)
    }, [open, onClose])

    if (!open || typeof document === 'undefined') return null

    const trapTab = (e: ReactKeyboardEvent) => {
      if (e.key !== 'Tab' || !sheetRef.current) return
      const els = Array.from(sheetRef.current.querySelectorAll<HTMLElement>(FOCUSABLE)).filter(
        (el) => !el.hasAttribute('disabled'),
      )
      if (els.length === 0) return
      const first = els[0]!
      const last = els[els.length - 1]!
      if (e.shiftKey && document.activeElement === first) {
        e.preventDefault()
        last.focus()
      } else if (!e.shiftKey && document.activeElement === last) {
        e.preventDefault()
        first.focus()
      }
    }

    return createPortal(
      <div className={styles.backdrop} onClick={onClose}>
        <div
          ref={sheetRef}
          className={className ? `${styles.sheet} ${className}` : styles.sheet}
          role="dialog"
          aria-modal="true"
          aria-label={ariaLabel}
          tabIndex={-1}
          onClick={(e) => e.stopPropagation()}
          onKeyDown={trapTab}
        >
          {children}
        </div>
      </div>,
      document.body,
    )
  }
  ```

  ```ts
  /** 공유 텍스트에서 유효한 8자 초대코드를 추출(없으면 null). 붙여넣기 자동 채움/제출용. */
  export function extractInviteCode(raw: string): string | null {
    const compact = normalizeInviteCode(raw)
    if (compact.length < 8) return null
    for (let i = 0; i + 8 <= compact.length; i++) {
      const window = compact.slice(i, i + 8)
      if (isValidInviteCode(window)) return window
    }
    return null
  }
  ```

  ```ts
  import { supabase } from '@/lib/supabase/client'
  import { useAuth } from '@/state/auth'

  type ProfilePatch = { display_name?: string; color?: string; expectedVersion: number }

  // 내 프로필(이름·색) 자가 수정 — profiles_self_update 허용. 낙관적 락: version 조건부 + version+1.
  export function useUpdateProfile(coupleId: string | null) {
    const { user } = useAuth()
    const qc = useQueryClient()
    const mutation = useMutation<void, Error, ProfilePatch>({
      mutationFn: async ({ display_name, color, expectedVersion }) => {
        if (!user) throw new Error('로그인이 필요해요.')
        const patch: Record<string, unknown> = { version: expectedVersion + 1 }
        if (display_name !== undefined) patch.display_name = display_name

          .eq('id', user.id)
          .eq('version', expectedVersion)
          .select('id')

        if (coupleId) void qc.invalidateQueries({ queryKey: ['profiles', coupleId] })
      },
    })
    return { updateProfile: mutation.mutateAsync, isPending: mutation.isPending, error: mutation.error }
  }
  ```

  ```ts
    a.click()
  ```
