# CLAUDE.md — love_place 코딩 규칙 (최상위)

> 로컬 Claude Code 세션이 이 저장소에서 코딩할 때 **항상 먼저 읽는** 계약서.
> 소스 오브 트루스: `여행관리앱_설계서.md` (이하 "설계서"). 이 문서는 설계서를 '구현 계약'으로 증류한 것이며 **설계서와 모순될 수 없다**. 충돌 시 설계서 + 아래 "확정 스택"이 우선한다.

---

## 1. 프로젝트 정체성

- **둘이 쓰는 여행 관리 웹앱.** 사용자 = 정확히 2명(나 + 여자친구). 멀티테넌시·회원 퍼널·권한 등급 없음. (설계서 §1)
- **공유가 기본값.** 데이터는 둘 다 보고 편집한다. 개인 영역(내 일정)만 색/소유자로 갈린다(숨김 아님 — §4.2).
- **마찰 최소화.** 위시 장소 저장 ≤ 3탭(설계서 §8 — 이 흐름에만 적용). 다른 흐름은 §8 흐름별 목표 탭 수.
- **iOS 감성을 웹으로 근사.** 둘 다 모바일 Safari/브라우저 URL로 사용. 모션·전환·햅틱(가능 범위)으로 네이티브 감성에 근접.
- **1급 요구사항 3종**(2인이라도 무시 금지, §1): ① 동시 편집 충돌 ② 사적 데이터(동선·사진) 보호 ③ 관계 종료 시 데이터 소유권/내보내기(§10.4).

## 2. 확정 스택 (변경·반론 금지)

설계서 §2의 '선행 게이트'는 **웹앱**으로 해소되었다(설계서 B안 백엔드 기반). 아래에 반하는 코드/제안 금지.

- **플랫폼**: React + Vite + TypeScript. **PWA**(홈화면 추가). 토스 인앱 ✗, SwiftUI 네이티브 ✗, Expo/RN ✗.
- **백엔드**: Supabase — Postgres + Auth + RLS + Realtime + Storage + Edge Functions.
- **서버리스 프록시**: Supabase Edge Functions(Deno). 카카오/Claude/길찾기 키를 서버 보관, 호출자 인증(Supabase JWT 검증), 레이트리밋 + 월 사용량 상한, 결과 캐싱. (설계서 §2.1, §10.1)
- **지도**: 카카오맵 JS SDK(웹)로 표시. 카카오 로컬 REST(키워드 검색)·카카오모빌리티/TMap(길찾기)는 **프록시 경유**.
- **AI 경로**: Anthropic API(Claude), 프록시 경유. structured outputs(strict `json_schema`), 장소 화이트리스트 검증, 영업시간 환각 금지, 결정론적 후처리, 폴백. (설계서 §5.6)
- **인증**: Supabase Auth(매직링크/OAuth). 커플 바인딩 = 1회용·만료 초대코드, 1:1.
- **상태/실시간**: Supabase Realtime로 공유 데이터 자동 전파. 서버 상태는 **TanStack Query** 권장.

**웹 트레이드오프**: iOS 웹푸시 제약(PWA 홈화면 + iOS 16.4+) → 설계서 §8 **인앱 활동 피드**를 1차 알림 수단으로 둔다.

## 3. 폴더 구조

```
src/
  pages/            # 라우트 단위 화면
  components/       # 재사용 UI
  lib/              # supabase client · kakao · anthropic-types · utils
  hooks/
  state/            # 클라이언트 상태(오프라인 큐 등)
  styles/
  __tests__/        # vitest 단위/통합
supabase/
  migrations/       # SQL 마이그레이션(RLS 포함)
  functions/<edge-fn>/  # Edge Function(Deno) 프록시
e2e/                # Playwright 비주얼 스모크
public/             # PWA manifest, 아이콘 등
```

- 데이터 모델: Postgres, **snake_case**. 공유 테이블 전부 `couple_id` + 감사/동기화 필드(`created_at, updated_at, created_by, updated_by, deleted_at, version`). 자세한 스키마는 `.ai-factory/02-data-model.md`.

## 4. 코딩 전 `.ai-factory/` 읽는 순서 (필수)

새 패킷/태스크 착수 전 **이 순서로** 읽는다. 각 파일이 해당 영역의 계약이다.

1. `.ai-factory/00-stack-decision.md` — 확정 스택과 그 근거(설계서 §2 해소).
2. `.ai-factory/01-spec.md` — 기능 스펙/요구사항.
3. `.ai-factory/02-data-model.md` — 테이블·컬럼·도출 규칙(Wish/Visit, region FK, 색 도출 등).
4. `.ai-factory/03-proxy-contract.md` — Edge Function 프록시 입출력 계약(인증·한도·캐싱·structured output 스키마).
5. `.ai-factory/04-tasks.md` — 패킷 계획(P0~P5)과 현재 태스크 ID·DoD.

> 파일명 번호 접두는 위 순서를 고정한다. 실제 파일명이 다르면 접두 번호와 주제로 매칭해 같은 순서를 지킨다.
> 추가로 `.claude/rules/`의 규칙 파일(있으면 전부)을 읽고 따른다. 이 문서와 `.claude/rules/`가 충돌하면 더 구체적인 쪽(rules)을 우선하되, 설계서·확정 스택에는 절대 반하지 않는다.

## 5. 핵심 규칙 (위반 = 즉시 거부)

1. **API 키 클라이언트 절대 금지.** 카카오·Claude·길찾기·GitHub 등 모든 외부 키는 **Edge Function 프록시에만** 둔다. 프록시는 호출자 JWT 검증 + 레이트리밋 + 월 사용량 상한 + 캐싱을 갖춘다. "키 숨김 ≠ 엔드포인트 보호." (§2.1, §10.1)
2. **RLS 전 테이블.** 모든 공유 테이블에 `couple_id = 호출자의 couple_id` 정책 + `visibility(PERSONAL|SHARED)` 다단계. 멤버십 정본 = `couples.user_a/user_b`(profiles.couple_id는 캐시). 멤버 ≤ 2는 앱 레이어 강제. (§10.2, §4.2)
3. **soft-delete.** 삭제는 `deleted_at`만 채운다(휴지통/복구 유예). 물리 삭제 금지. 조회는 `deleted_at IS NULL` 필터. (§4.3)
4. **낙관적 락(version).** update는 `version` 조건부로. 서버 버전이 더 높으면 **충돌 감지**(LWW·무음 덮어쓰기 금지). memo·priority 등 손실 아까운 필드는 머지/append 고려. (§4.3)
5. **오프라인 큐잉.** 쓰기는 로컬 큐 → 재연결 시 동기화, 충돌은 사용자에게 표시. P1부터 테스트 포함. (§4.3, §9.2)
6. **색만으로 상태 구분 금지.** 트랙/상태는 **색 + 패턴/라벨로 이중화**(색각 이상 대응). VoiceOver 라벨, Dynamic Type, Reduce Motion, 다크모드 기본 지원. (§8)
7. **상태는 도출, 저장 아님.** "가고싶음 = wishes 존재", "가봤음 = visits 존재". `place.status` 같은 플래그 만들지 말 것(불일치 원천 차단). 캘린더 트랙 색은 런타임 도출(SHARED=퍼플 / PERSONAL=소유자색). (§4.2)
8. **AI 출력 신뢰 금지.** structured output strict + zod/ajv 검증 + 장소 화이트리스트(입력 place_id 집합 밖이면 거부) + 영업시간 환각 금지 + 도착시각 앱 재계산 + 폴백. (§5.6)
9. **사진 발행 = EXIF 스트립.** 블로그/공개 발행 시 원본 복사 → EXIF/GPS 제거 → 리사이즈 → 공개 경로 재업로드. 비공개 서명 URL 그대로 넘기지 않는다. 비공개가 기본값. (§7, §10)
10. **TS strict.** `tsc`가 0 에러여야 한다. `any` 남발 금지. 외부 응답은 타입 검증 후 사용.

## 6. 제출 전 게이트 체크리스트 (전부 통과해야 완료)

설계서 §해당 절을 근거로 인용하며, 아래를 **모두** 통과하지 못하면 태스크 미완료다.

- [ ] **`tsc` 0 에러** — `npm run typecheck`(또는 `tsc --noEmit`).
- [ ] **vitest 통과** — 단위/통합 테스트, 특히 동기화·충돌·오프라인 시나리오(P1부터). `npm run test`.
- [ ] **build 성공** — `npm run build`(Vite 프로덕션 빌드).
- [ ] **Playwright 비주얼 스모크** — `e2e/` 핵심 화면 렌더/스냅샷. `npm run e2e`(또는 정의된 스크립트).
- [ ] **RLS 격리 확인** — 다른 couple의 행이 보이지 않고 쓰지 못함을 마이그레이션/테스트로 검증(§10.2). 새 공유 테이블엔 RLS 정책 필수.
- [ ] **EXIF 스트립 확인** — 발행 경로 변경 시 가공본에 GPS/EXIF가 남지 않음을 확인(§7). 발행 미관련 변경이면 N/A 명시.
- [ ] **접근성 회귀 없음** — 색+패턴 이중화, VoiceOver 라벨, Reduce Motion 유지(§8).

> 스크립트명이 `package.json`과 다르면 실제 스크립트로 대체하되 위 7개 게이트의 **의도**를 모두 충족한다.

## 7. 테스트·비주얼 검증 방침

- **무엇을 테스트하나 (P1부터 횡단, §9.2)**: ① 동기화(Realtime 전파) ② 충돌(낙관적 락 — 서버 버전↑일 때 거부/표시) ③ 오프라인(약전파 큐잉 → 재연결 동기화, **유실 0**) ④ RLS 격리 ⑤ AI 경로의 스키마 검증·화이트리스트·폴백(P4).
- **단위/통합 = vitest** (`src/__tests__/`). 도출 로직(Wish/Visit→상태, 색 도출, 도착시각 재계산), 프록시 입력 검증, 오프라인 큐 머지를 우선 커버.
- **E2E·비주얼 = Playwright** (`e2e/`). 핵심 화면(지도/일정/장소/추천/우리)의 렌더·빈 상태·로딩 상태 스모크. 모바일 뷰포트(모바일 Safari 근사) 기준. 다층 빈 상태·로딩 디테일도 캡처(§8).
- **빈 상태/로딩 필수.** 모든 리스트/탭은 빈 상태와 로딩 상태를 가진다(다층 빈 상태, 콜드스타트 대비 — §5.6, §8). 누락 시 미완료로 간주.
- **수동 확인은 보조.** 게이트가 1차. 새 동작은 가능하면 자동 테스트로 못 박는다.

## 8. 패킷 계획 (참조용 — 정본은 `.ai-factory/04-tasks.md`)

- **P0 토대**: 스캐폴드, Supabase·Auth, couples/profiles + 초대·연결, RLS 골격, Edge Function 프록시 스켈레톤, 내보내기 v0(JSON), CI.
- **P1 MVP**: 위시 장소(카카오 자동완성 디바운스 250ms→프록시, 중복=kakao_place_id, stale 취소토큰) + 지도 별표 마커 + 공유(Realtime) + 충돌/오프라인 기본.
- **P2 일정**: 3트랙 공유 캘린더(월/주/일/아젠다, 종일/반복 RRULE/타임존, 장소 연결, 사용자별 리마인더, 색 도출).
- **P3 기록**: 가본 장소 + trips + 공유 사진 앨범(EXIF 제안 + 수동 확정, 미분류 트레이, 썸네일 지연로딩).
- **P4 차별화**: 지역별 추천(임계치 3~5, 콜드스타트 시드) + AI 경로 → 지도 폴리라인·"함께 캘린더" 자동 이벤트(itinerary_id 출처).
- **P5 마감**: blog auto writer 연동(가공본·비공개 기본) + UX 폴리시·애니메이션·접근성.

**횡단(모든 단계)**: 테스트, 백업/내보내기 점진강화(§10.4 관계종료 ZIP+JSON 양측 동등), 비용 추적(TCO), 보안/프라이버시 체크.
