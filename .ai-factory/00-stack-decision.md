# 00 — 스택 결정 (선행 게이트 해소 기록)

> 소스 오브 트루스: `여행관리앱_설계서.md`. 이 문서는 설계서 §2 '선행 게이트'를 **웹앱**으로 확정한 기록 + 사용자 공급물 체크리스트 + 출시 전 재확인 항목이다. 설계서와 모순되는 어떤 결정도 금지.

## D5 결정 (2026-06 확정): 지도 표시·장소 검색 = 네이버 / 결정·근거·롤백 정책(카카오 코드 보존)

**정본:** 지도 표시 = **네이버 지도 JS SDK v3**(NCP Web Dynamic Map), 장소 검색·자동완성 = **네이버 지역검색 API**(프록시 경유). **근거:** 한국 장소 데이터·좌표 품질과 공식 웹 JS SDK(MarkerClustering 샘플 제공)로 클러스터링·커스텀 마커가 자연스러움. **롤백 정책:** 기존 카카오 코드(카카오맵 JS·카카오 로컬 키워드 검색)는 **삭제하지 않고 롤백용으로 보존**하되 현재 **미사용**이다. 따라서 코드 식별자(`kakao_place_id` 컬럼, `uq_places_couple_kakao` 인덱스, `kakao-search` Edge Function 함수명, `KakaoSearchReq/Res/PlaceHit`·`kakaoPlaceId` 타입/필드, Playwright testid 토큰)는 **이름을 그대로 유지**하고 의미만 네이버로 해석한다(네이버는 고유 장소 ID가 없어 `kakao_place_id` 값에 `norm(name)|norm(address)` 합성키를 저장 — `src/lib/naver/normalize.ts`). **길찾기(route-eta·카카오모빌리티/TMap)는 D5 스코프 밖 — 카카오/TMap 유지.** 이하 다른 문서의 'D5' 참조는 이 결정을 가리킨다.

## 1. 결정 — 웹앱 (React + Vite + TS + Supabase + 네이버 지도 JS + Edge Functions)

설계서 §2는 두 길(A안 SwiftUI+CloudKit / B안 Expo·RN+Supabase)을 두고 "스택을 먼저 정해야 §5.4·§7·§8이 공회전하지 않는다"고 했다. 본 프로젝트는 **설계서 B안 백엔드(Supabase)를 채택하되, 프런트엔드 플랫폼을 RN/Expo가 아니라 웹앱(React + Vite + TypeScript, PWA)으로 구체화**한다.

### 확정 스택
- **플랫폼:** 웹앱 = React + Vite + TypeScript, **PWA(홈화면 추가)**. 두 사람이 모바일 Safari/브라우저 URL로 사용. (토스 인앱 아님 · SwiftUI 네이티브 아님 · Expo/RN 아님.)
- **백엔드:** Supabase — Postgres + Auth + RLS + Realtime + Storage + Edge Functions. (설계서 §2 B안 백엔드.)
- **서버리스 프록시:** Supabase Edge Functions(Deno) — 카카오/Claude/길찾기 키를 **서버 보관**, 호출자 인증(Supabase JWT 검증), 레이트리밋 + 월 사용량 상한, 결과 캐싱(§2.1). 1차는 Edge Functions; 지연 민감 호출(자동완성)에서 콜드스타트가 문제되면 Vercel/CF Workers를 대안으로 검토(§2.1 런타임 선택).
- **지도:** **네이버 지도 JS SDK v3(웹, 정본 — D5)** 로 표시(카카오맵 롤백 보존). 장소 검색·자동완성 = 네이버 지역검색(프록시 경유); 길찾기는 카카오모빌리티/TMap 유지(프록시 경유, §6).
- **AI 경로:** Anthropic API(Claude) **프록시 경유** — structured outputs(strict `json_schema`), 장소 화이트리스트 검증(입력 `place_id` 집합 밖이면 거부), 영업시간 환각 금지, 결정론적 후처리(도착시각=직전+체류+이동, 앱이 재계산), 폴백(좌표 TSP 순서 + 끼니 슬롯)(§5.6).
- **인증:** Supabase Auth(매직링크 또는 OAuth). 2인이라 단순. 커플 바인딩 = **1회용·만료 초대코드**(§10.3).
- **상태/실시간:** Supabase Realtime로 공유 데이터 자동 전파. 서버 상태는 **TanStack Query** 권장.

### 설계서 §2 A/B 대비 '웹' 선택 근거·트레이드오프

설계서의 A안/B안 비교는 둘 다 **네이티브/RN 클라이언트**를 전제했다. 웹앱은 **B안의 변형**(같은 Supabase 백엔드, 클라이언트만 웹)으로, 설계서 §2 분기 표의 다섯 축에서 B안의 장점을 대부분 승계하면서 RN 지도 연동 난이도(설계서 §5.5의 "A안 선택 실질 근거")를 네이버 지도 **웹 JS SDK v3**(정본 — D5)로 우회한다.

| 결정 축 (설계서 §2 분기 표) | 웹앱(채택)에서 | 근거 / 트레이드오프 |
|---|---|---|
| 사진 블로그 핸드오프(§7) | Supabase Storage 공개/서명 URL로 자연스러움 | B안 이점 그대로 승계 — CKAsset의 "공개 URL 없음" 문제 회피 |
| 서버 측 처리(§5.6 AI·§7) | Edge Function에서 서버 처리 가능 | B안 이점 승계 — 화이트리스트 검증·EXIF 스트립·재업로드를 서버에서 |
| 지도 UX(§5.5) | **네이버 지도 JS SDK = 공식 웹 SDK(MarkerClustering 샘플)** | **웹의 결정적 이점.** RN은 공식 SDK 없어 WebView/네이티브 모듈 의존(§5.5)이었으나, 웹은 네이버 지도 JS SDK v3가 정식 지원이라 클러스터링(MarkerClustering 샘플)·커스텀 마커가 자연스러움(정본 — D5) |
| 프라이버시(§10) | RLS로 `couple_id` 통제(설계 필요) | B안과 동일 — CloudKit "둘의 iCloud에 머묾"의 기본 우위는 포기, RLS+소프트삭제로 보완 |
| 학습/속도 | 기존 JS/React 역량 즉시 재사용, Swift·Xcode·Mac 불필요 | B안 이점 승계 + Expo Dev Build/네이티브 빌드 단계도 제거 |

**웹 고유 트레이드오프(설계서 가정 대비):**
- **iOS 네이티브 감성 = 근사치.** 제스처/햅틱/전환은 모션·CSS 전환으로 *보완*하되 진짜 네이티브는 아님(설계서 §1·§8의 "iOS 네이티브 감성" 목표는 근사치로 수용).
- **iOS 웹푸시 제약.** iOS 웹푸시는 **PWA 홈화면 추가 + iOS 16.4+** 에서만 동작 → 설계서 §8 **'인앱 활동 피드'를 1차 알림 수단**으로 삼고, 웹푸시는 가능한 단말에서만 보조로(설계서 §8 "거부 시 인앱 피드 폴백"과 일관).
- **이점(왜 웹이 이 프로젝트에 맞나):** App Store 심사·**$99 불필요**, TestFlight/Expo 빌드 파이프라인 불필요, **URL 즉시 사용**(둘이 링크만 열면 끝), 네이버 지도 웹 JS SDK가 RN보다 쉬움(§5.5), Storage 공개/서명 URL로 사진·블로그가 자연스러움(§5.4·§7).

> 설계서 §2의 데이터 모델·기능·UX 계약은 플랫폼 무관하게 **그대로 적용**된다(설계서 §2 말미). 웹 선택은 클라이언트 구현 디테일만 바꾼다.

## 2. 사용자가 직접 공급해야 할 것 (체크리스트)

> 키는 **클라이언트에 절대 박지 않는다**(§10.1). 네이버 지도 JS SDK용 **네이버 Client ID**(`ncpKeyId`, `VITE_NAVER_MAP_CLIENT_ID`, 도메인 제한)만 도메인 제한과 함께 클라이언트 노출이 불가피하며(웹 SDK 특성), 그 외 REST/시크릿 키는 전부 Edge Function 환경변수(Supabase secrets)에 보관한다.

- [ ] **네이버 클라우드 플랫폼 / 네이버 개발자 (정본 — D5)**
  - [ ] **네이버 지도 Client ID** (`ncpKeyId`, `VITE_NAVER_MAP_CLIENT_ID`) — 네이버 지도 JS SDK v3(NCP Web Dynamic Map, 웹 지도 표시)용. *서비스 환경 > Web 도메인 등록* + 도메인 제한 필수.
  - [ ] **네이버 지역검색 키** (`X-Naver-Client-Id`/`X-Naver-Client-Secret`) — 네이버 지역검색(자동완성, §5.2)용. **Edge Function secret에만** 저장.
  - [ ] NCP Web Dynamic Map·지역검색 API 사용 설정(콘솔에서 활성화).
- [ ] **카카오 개발자 앱 (developers.kakao.com)** — *(롤백 보존, 미사용)*
  - [ ] **JavaScript 키** — 카카오맵 JS SDK용 *(롤백 보존, 미사용 — D5로 네이버 지도 SDK가 정본)*. *플랫폼 > Web 도메인 등록* + 사이트 도메인 제한.
  - [ ] **REST API 키** — 카카오 로컬 키워드 검색용 *(롤백 보존, 미사용 — D5로 네이버 지역검색이 정본)*. **Edge Function secret에만** 저장.
- [ ] **Supabase 프로젝트 (supabase.com)**
  - [ ] **프로젝트 URL** (`SUPABASE_URL`).
  - [ ] **anon key** — 클라이언트용(RLS로 보호되므로 노출 허용).
  - [ ] **service_role key** — Edge Function 서버 전용. **클라이언트 절대 금지.**
- [ ] **Anthropic API 키 (console.anthropic.com)** — AI 경로 생성(§5.6)용. **Edge Function secret에만** 저장. 월 사용량 상한을 콘솔과 프록시 양쪽에서 설정.
- [ ] **길찾기 키 — 카카오모빌리티 *또는* TMap** (§5.6·§6)
  - [ ] 카카오모빌리티 자동차 길찾기 REST 키, **또는** TMap(SK open API) 경로안내 키. 둘 중 하나 택1. **Edge Function secret에만** 저장.
- [ ] **(선택) blog auto writer의 실제 형태** — 설계서 §7은 *형태 확정 후* 운반책 한 가지로 좁힌다. 아래 중 무엇인지 알려줄 것:
  - [ ] **REST/Webhook** 웹 서비스인가? → `POST /import`(고정 페이로드, §7) + 엔드포인트 URL/인증 토큰.
  - [ ] **Jekyll/정적 사이트(GitHub Pages 계열)** 인가? → GitHub repo + 토큰(프록시 보관), `_drafts` 푸시 경로.
  - [ ] **iOS 단축어(Shortcuts)/공유 시트** 인가? → 코딩 최소(§7 MVP 1순위 권장), 페이로드만 넘김.
  - 형태가 확정되기 전까지 §7은 **고정 페이로드 스키마**만 구현하고 운반책은 보류.

## 3. 출시 전 재확인 항목 (설계서가 "공식 페이지에서 확인"이라 명시한 것들)

> 설계서 §5.5·§5.2·§6·§9.2는 무료 한도가 정책 변동 영향을 받으므로 **출시 직전 공식 페이지 재확인**을 못박았다. 아래는 그 체크리스트(현 문서 시점 추정은 참고용, 출시 전 1차 출처 확인 필수).

- [ ] **네이버 지도 JS SDK(NCP Web Dynamic Map) 2026 한도·과금** — 네이버 클라우드 플랫폼 공식 가격/정책 페이지에서 현재 뷰 단위 종량·무료 제공량 확인(정본 — D5). 네이버 모바일 동적 지도는 무료 제공량이 없는 것으로 알려짐(2023 개편·2025-03 무료 중단 예고 → 종량)이나 출시 전 재확인.
- [ ] **카카오맵 JS SDK — 참고용 비교만 (롤백 보존, 미사용).** 카카오 공식 가격/정책 페이지에서 무료 호출/뷰 한도 확인. (설계서 §5.5: 과거 카카오맵 1순위 서술은 결정 근거 보존용 — D5로 네이버가 정본. 카카오 코드는 롤백용 보존, 현재 미사용.)
- [ ] **네이버 지역검색 일/월 쿼터** — 네이버 개발자 콘솔에서 현재 일/월 한도 확인(§5.2·§6, 정본 — D5). 2인 규모는 무료 안일 가능성 높으나 출시 전 확정.
- [ ] **카카오모빌리티 / TMap 길찾기 한도** — 카카오모빌리티 자동차 길찾기 ≈ 일 1만 건, TMap 경로안내 ≈ 일 1,000건 수준(설계서 §5.6 추정). 채택한 쪽의 현재 한도 재확인.
- [ ] **iOS 웹푸시 제약** — PWA **홈화면 추가 + iOS 16.4+** 에서만 웹푸시 가능 재확인. 미충족 단말은 §8 인앱 활동 피드로 폴백(이 문서 §1 트레이드오프).
- [ ] **(웹 특이) 네이버 지도 Client ID(`ncpKeyId`) 도메인 제한** — 배포 도메인을 NCP 콘솔 Web 서비스 환경에 등록했는지, 키 도용 방지용 도메인 화이트리스트가 걸렸는지 확인(정본 — D5).

## 5. 이 결정이 잠그는 후속 계약 (요약)

- 모든 외부 키 → Edge Function 프록시(인증·레이트리밋·월 상한·캐싱), 네이버 지도 Client ID(`ncpKeyId`)만 도메인 제한부 클라이언트 노출(정본 — D5; §2.1·§10.1).
- 전 공유 테이블 RLS(`couple_id` 기준) + 소프트삭제(`deleted_at`) + 낙관적 락(`version`) + 오프라인 큐(§4.3·§10.2).
- 알림 1차 수단 = 인앱 활동 피드, 웹푸시는 iOS 16.4+ PWA 보조(§8).
- 색만으로 상태 구분 금지(색+패턴/라벨), VoiceOver/Dynamic Type/Reduce Motion/다크모드 기본(§8).
- 게이트: tsc 0, vitest, build, Playwright 비주얼 스모크.
