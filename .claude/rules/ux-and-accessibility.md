# Rule: UX & 접근성 (구현 계약)

> 근거: 설계서 §8(UX 디테일), §1(설계 원칙·탭3번), §5.1(캘린더 색·칩), §5.4(미분류 트레이), §5.5(출처 아바타), §5.6(콜드스타트·다층 빈상태), §10.3(상호 동의 온보딩).
> 스택 결정: **웹앱(React + Vite + TS, PWA)**. iOS 네이티브가 아니므로 네이티브 감성은 **근사치**로 구현하고, iOS 웹푸시 제약 때문에 **인앱 활동 피드를 1차 알림 수단**으로 둔다. 이 결정에 반하는 구현 금지.

---

## 1. iOS 감성 근사 (웹에서의 네이티브 느낌)

둘 다 모바일 Safari로 쓴다. 네이티브 SwiftUI가 아니므로 **모션·전환·터치 피드백으로 격차를 메운다.**

- **전환·모션**: 페이지/모달 전환은 부드러운 spring/ease 전환. 라우트 전환·바텀시트·탭 전환에 일관된 모션 토큰 사용. CSS `transition`/`transform` 기반(레이아웃 thrash 피하고 `transform`·`opacity`만 애니메이트).
- **터치 타깃**: 최소 44×44px(HIG). 하단 탭바·주요 액션 버튼은 엄지 도달 영역.
- **제스처(가능 범위)**: 당겨서 새로고침, 스와이프(목록 액션), 롱프레스 컨텍스트 메뉴는 **웹에서 가능한 만큼만** 구현하고 항상 **버튼 대체 경로**를 함께 둔다(제스처는 발견성이 낮음).
- **햅틱(가능 시)**: 지원 단말에서 `navigator.vibrate()`로 가벼운 햅틱(장소 저장·하트·마커 드롭 성공 시). **iOS Safari는 `vibrate` 미지원**이므로 **있으면 좋은 보강일 뿐 핵심 피드백을 햅틱에만 의존 금지**(시각/모션 피드백을 항상 병행).
- **시그니처 마이크로인터랙션**: 위시 저장 시 하트 애니메이션, 지도 마커 드롭, 탭 전환 모션. 단 §5의 Reduce Motion에서는 생략(아래 5장).
- **`-webkit-fill-available`·safe-area**: `env(safe-area-inset-*)`로 노치/홈인디케이터 회피. iOS 100vh 버그 회피.

## 2. 둘만의 경험 — 출처 아바타 · ❤️ 리액션

- **출처 아바타(전 항목)**: 모든 공유 항목(장소·사진·방문기록·이벤트)에 **누가 추가/편집했는지** 작은 아바타로 출처 표시(`createdBy`/`updatedBy` + `profiles.avatar_url`/`color`). 지도 마커에도 작은 아바타 점으로 출처(§5.5).
- **가벼운 ❤️ 리액션**: 장소·사진·방문기록(`reactions.target_type = PLACE|PHOTO|VISIT`)에 가벼운 이모지 리액션 허용. **무거운 소셜 기능(댓글 스레드·팔로우 등) 금지.**
- **하트 ≠ 리액션 (혼동 금지)**: 위시 우선순위 하트는 `wishes.priority`(개인 의도의 세기), ❤️ 리액션은 `reactions`(누가 눌렀나). UI에서도 시각적으로 구분(우선순위 = 위시 카드의 하트 슬라이더/단계, 리액션 = 항목 하단 이모지 칩). 한 컨트롤로 섞지 말 것.

## 3. "탭 3번" 원칙의 적용 범위 (흐름별 목표 탭 수)

"탭 3번"은 **위시 장소 저장에만** 적용한다. 다단계 흐름엔 별도 목표를 둔다.

| 흐름 | 목표 | 측정 기준 |
|---|---|---|
| **위시 장소 저장** | **≤ 3탭** (원칙) | 검색 입력 → 후보 탭 → 저장. (메모·태그·하트는 선택, 탭 수에 미포함) |
| 가본 곳으로 전환(날짜·여행 선택) | ≤ 5탭 | 장소 상세 → "다녀왔어요" → 날짜 → 여행 선택 → 확정 |
| AI 코스 생성 → 캘린더 삽입 | 가이드형 다단계 | 탭 수 대신 **단계별 명확성**(현재 단계/남은 단계 표시) |

- 위시 저장 흐름은 회귀 테스트로 탭 수를 고정(3 초과 시 실패). DoD: 저장 ≤3탭 & 자동완성 체감 ≤400ms(§9 단계1).

## 4. 접근성 — 색+패턴/라벨 이중화, VoiceOver, Dynamic Type, 다크모드

**색만으로 상태/트랙을 구분 금지(색각 이상 대응).** 항상 색 + 패턴/라벨/아이콘으로 이중화한다.

- **캘린더 트랙(§5.1)**: 색만 다르면 안 됨. SHARED=함께(퍼플), PERSONAL 내=블루, PERSONAL 상대=핑크 — 각 트랙에 **라벨 텍스트 + 패턴/아이콘**(예: 함께=●, 내=▲, 상대=■, 또는 텍스트 칩) 병행. 상단 트랙 칩에도 색+텍스트.
- **지도 마커(§5.5)**: 가고싶음=빈 별, 가봤음=채운 별 **+ 체크 아이콘**. 색 차이에만 의존 금지(모양으로 구분되게).
- **상태/배지**: 사진 '자동' 배지, 미분류 트레이, 충돌 표시 등은 **아이콘+텍스트**로.
- **VoiceOver / 스크린리더**: 의미 있는 모든 요소에 접근성 라벨. 시맨틱 HTML 우선(`button`/`nav`/`main`/`h1..`/`ul`). 아바타·마커·리액션 등 아이콘 전용 컨트롤은 `aria-label`. 동적 변경(Realtime 갱신, 충돌 발생)은 `aria-live` 영역으로 안내. 모달/바텀시트는 포커스 트랩 + `role="dialog"` + ESC/닫기.
- **Dynamic Type(큰 글자)**: 폰트 크기는 `rem`/상대단위, 컨테이너는 텍스트 확대에 깨지지 않게(줄바꿈 허용, 고정 높이 텍스트 박스 금지). 사용자 글자 크기 설정 존중(고정 px 폰트 금지).
- **다크 모드(기본 지원)**: `prefers-color-scheme` 대응. 색 토큰을 CSS 변수로 두고 라이트/다크 양쪽 정의. 트랙 색(블루/핑크/퍼플)도 다크에서 대비(WCAG AA, 본문 4.5:1) 유지.
- **대비·포커스**: 텍스트 대비 WCAG AA. 키보드 포커스 링 항상 가시(`:focus-visible`).

## 5. Reduce Motion (모션 축소)

`prefers-reduced-motion: reduce`를 기본 존중한다.

- 카운트업(숫자 애니메이션), 페이지/마커 드롭·하트 펄스 등 **장식 모션을 생략**하거나 즉시 최종 상태로 전환.
- 전환은 fade/즉시로 대체(슬라이드·스프링 제거). 패럴랙스·자동재생 모션 금지.
- 구현: 모션 토큰을 미디어쿼리로 분기(`@media (prefers-reduced-motion: reduce) { ... transition: none; }`), 또는 JS 애니메이션은 `matchMedia('(prefers-reduced-motion: reduce)')` 확인 후 스킵.

## 6. 알림 권한 & 인앱 활동 피드 폴백 (웹푸시 제약 연결)

**웹앱 제약**: iOS 웹푸시는 **PWA 홈화면 추가 + iOS 16.4+**에서만 동작. 브라우저 탭 상태에선 푸시 불가. 따라서:

- **권한은 맥락에서 요청**: 첫 실행에 무턱대고 Notification 권한 요청 금지. **첫 리마인더 설정 같은 맥락**에서만 요청(요청 전 왜 필요한지 1줄 설명).
- **거부/미지원 시 인앱 활동 피드로 폴백(1차 알림 수단)**: "상대가 새 장소를 추가했어요", "○○ 여행 D-3" 같은 둘 사이 신호를 **앱 안 활동 피드**에서 보여준다(Realtime 기반). 웹푸시가 안 되거나 거부돼도 신호를 놓치지 않게.
- **알림 범위 한정**: 일정 리마인더 + 둘 사이 신호(상대가 추가/편집, 여행 D-day)로 한정. 마케팅/소셜성 알림 금지.
- **PWA 안내**: 푸시를 원하는 사용자에겐 "홈 화면에 추가" 안내를 부드럽게 제시(강요 금지). 미설치/미지원이어도 핵심 경험(인앱 피드)은 그대로 동작.

## 7. 온보딩 최소화 & 다층 빈 상태

- **온보딩 3스텝으로 끝**: ① 상대 초대/연결(1회용 만료 코드, §10.3) → ② 캘린더 색상 두 개 고르기 → ③ **위치/사진 공유 상호 동의**(§10.3). 그 외 튜토리얼·강제 투어 금지.
- **빈 상태는 친근하게 + 행동 유도**: "첫 가고싶은 장소를 추가해보세요" 식. 빈 화면을 죽은 화면으로 두지 말 것.
- **다층 빈 상태(데이터 의존 탭)**:
  - **추천 탭**: 데이터 없을 때 죽은 탭이 되지 않게 **회고형("지난 ○○ 여행 다시 보기")·시드 추천**으로 채움. 지역별 위시가 임계치(3~5) 미만이면 "○개 더 모으면 코스를 짜드려요" 진행 안내(§5.6 콜드스타트).
  - **지도/장소/기록**: 초기엔 첫 액션 CTA, 데이터가 일부 쌓이면 부분 빈 상태(예: 사진 미분류 트레이만 비었을 때) 구분.
- **로딩 디테일 필수**: 스켈레톤/플레이스홀더(빈 회색 박스 X). 사진 그리드는 썸네일 지연 로딩 + 블러업(§5.4). 자동완성은 로딩 인디케이터 + stale 응답 무시.
- **에러 상태**: 인라인 에러 + 재시도 + 입력값 보존(§5.2 엣지케이스). 오프라인/충돌은 명시적 배지·표시(§4.3).

## 8. 구현 게이트 (이 규칙의 검증)

- 위시 저장 흐름 탭 수 ≤3 회귀 테스트.
- 색 외 구분(패턴/라벨) — 트랙·마커·배지에 텍스트/아이콘 동반 단위 점검.
- `prefers-reduced-motion`·`prefers-color-scheme` 분기 동작 확인.
- 알림 거부 시 인앱 피드 폴백 경로 동작.
- Playwright 비주얼 스모크에 라이트/다크 + 빈 상태/로딩 상태 포함.
