# Rule: 보안 · 프라이버시 (설계서 §10)

> love_place는 둘만의 사적 데이터(사진·동선·일정)를 다룬다. 보호는 1급 요구사항이다.
> 이 문서는 설계서 §10(+§2.1, §4.3, §7, §8.3)을 **구현 계약**으로 증류한 것이다. 스파인(웹앱=React+Vite+TS / Supabase B안)과 모순 금지.

---

## 0. 절대 규칙 (위반 시 머지 불가)

1. **API 키는 클라이언트에 절대 박지 않는다.** 카카오(로컬·맵·모빌리티/TMap)·Anthropic·길찾기·GitHub 토큰 전부 Edge Function 프록시에 보관. 클라이언트 번들·env·소스에 키 문자열이 들어가면 안 됨. (§10.1, §2.1)
2. **모든 공유 테이블에 RLS.** 정책 없는 공유 테이블 = 머지 불가. 기준 컬럼은 `couple_id`. (§10.2)
3. **블로그 발행 외 모든 데이터는 비공개가 기본값.** 공개로 가는 유일한 경로는 §3의 가공 파이프라인뿐. 원본/비공개 서명 URL을 외부로 노출 금지. (§7, §10.3)
4. **삭제는 soft-delete(휴지통).** 물리 삭제는 관계 종료(§5)·복구 유예 경과·법적 삭제 요청 처리에서만. (§4.3)
5. **색만으로 상태/소유자 구분 금지** — 색 + 패턴/라벨 이중화(이 문서 범위 밖이지만 프라이버시 라벨에도 적용). (§8.3)

## 1. 키 · 엔드포인트 (§10.1, §2.1)

모든 외부 호출은 `앱 → Supabase Edge Function 프록시 → 카카오/Claude/길찾기` 순서다. "키 숨김 ≠ 엔드포인트 보호" — 인증 없는 프록시는 오픈 프록시가 되어 카카오 쿼터 소진·**내 명의 Anthropic 과금 폭탄**(토큰당 과금)을 일으킨다.

### 모든 프록시 Edge Function이 반드시 갖출 것
- **호출자 인증**: 요청의 `Authorization: Bearer <supabase_jwt>`를 검증. 미인증 → `401`. 검증된 `user.id`가 호출자의 `couple_id` 멤버인지 확인(`couples.user_a/user_b`).
- **레이트리밋**: 엔드포인트별 분당/일당 한도. 초과 → `429` + 친절한 거절 메시지. 자동완성처럼 빈번한 호출과 AI 경로처럼 비싼 호출은 한도를 분리.
- **월 사용량 상한(비용 가드)**: 특히 Claude. 누적 비용/호출 수가 월 상한 초과 → 거절(`429`/`402` 류 + 사용자 친화 메시지). 상한은 환경설정 값으로.
- **결과 캐싱**: 동일 입력(같은 place 집합·제약)의 AI 코스 캐싱, 자동완성은 짧은 TTL 캐시. 비용·지연 절감.
- **(선택·강화)** iOS App Attest/DeviceCheck 류는 웹앱이라 직접 적용 불가 → JWT + Origin/CORS 화이트리스트 + 레이트리밋으로 대체.

- [ ] 새 외부 API 호출이 클라이언트가 아닌 Edge Function에서 일어나는가?
- [ ] 그 Edge Function이 JWT 검증 + 레이트리밋 + 월상한을 통과하는가?
- [ ] 키가 Supabase Function Secrets(서버)에만 있고 `import.meta.env`(클라)에는 없는가?
- [ ] CORS Origin 화이트리스트가 우리 앱 도메인으로 제한돼 있는가?

## 2. 데이터 격리 — RLS (§10.2)

모든 공유 테이블(`couples, profiles, places, wishes, visits, trips, photos, events, itineraries, reactions, regions`*)에 RLS를 켜고 `couple_id = 호출자의 couple_id` 정책을 건다. "2인이라 간단"이 아니라 **공유/개인 다단계 정책**을 명시적으로 설계·테스트한다.

(*`regions`는 마스터/공용일 수 있으나 읽기 정책은 명시할 것. `couples`는 `user_a/user_b` 멤버십으로 정책을 건다.)

### 정책 골격
- **couple_id 격리**: 호출자가 속한 couple의 행만 `select/insert/update/delete`. 호출자의 `couple_id`는 `auth.uid()` → `couples`(또는 `profiles` 캐시, 어긋나면 `couples` 신뢰, §4.2) 조회로 도출.
- **visibility 다단계 (events)**: `visibility=SHARED`는 커플 양쪽 모두 접근. `PERSONAL`도 한 캘린더 공유라 양쪽이 보되 색만 소유자로 갈림(§4.2) — 즉 PERSONAL을 상대에게 숨기는 RLS는 기본 범위 밖. (진짜 비공개가 필요하면 별도 플래그로 확장하되 그땐 RLS로 owner-only를 명시.)
- **soft-delete 가시성**: 기본 쿼리는 `deleted_at IS NULL`만 노출. 휴지통/복구 화면만 삭제된 행을 본다.
- **insert 시 couple_id 위조 차단**: `with check (couple_id = 호출자 couple_id)`로 남의 couple에 쓰기 금지.

### 격리 테스트 (필수 — §9.2 보안 트랙, P0 DoD "RLS 격리 확인")
`src/__tests__` 또는 Supabase 통합 테스트에 **다른 couple의 데이터에 접근 불가**를 검증하는 케이스를 둔다:
- [ ] couple A의 JWT로 couple B의 place/event/photo `select` → 0건
- [ ] couple A의 JWT로 couple B 행 `update/delete` → 0행 영향(또는 거부)
- [ ] couple A의 JWT로 `insert` 시 `couple_id`를 B로 위조 → 거부
- [ ] 미인증 요청 → 전부 거부
이 테스트는 P0부터 존재하고 CI 게이트에 포함된다.

## 3. 사진 · 위치 — 권한 · 동의 · 블로그 발행 (§7, §10.3, §8)

### 3.1 권한 요청 (정직하게, 맥락에서)
- 첫 실행에 무턱대고 요청하지 않는다. **맥락에서 요청**(예: 사진 업로드 시점에 사진 권한, 위치 기반 기능 시점에 위치). (§8 알림 권한 패턴과 동일 철학)
- 거부해도 앱이 동작하도록 폴백(사진 없이 텍스트, 위치 없이 수동 핀).

### 3.2 온보딩 상호 동의 (§10.3, §8 온보딩)
상대의 위치·사진을 다루므로 온보딩에서 **상호 동의**를 받는다:
- [ ] "서로의 위치 정보를 공유·표시하는 데 동의" (위치정보보호법 관점)
- [ ] "서로 업로드한 사진을 공유 앨범에서 함께 보는 데 동의"
동의 여부/시각을 기록(감사). PIPA·위치정보보호법 관점에서 2인용이라도 **동의·최소수집** 원칙을 따른다(필요 이상 수집·보관 금지).

### 3.3 블로그 발행 = 명시적 공개 (§7, §10.3) — 절대 규칙
블로그로 보내는 것만이 유일한 공개 행위다. 발행 시 **반드시** 다음 파이프라인을 거친다(프록시/Edge Function에서):

```
원본 사진 (비공개 Storage)
  → 복사본 생성
  → EXIF/GPS 메타데이터 스트립   ← 집·동선 유출 차단
  → 리사이즈
  → 공개 경로에 새로 업로드        ← 별도 공개 버킷/경로
  → payload.photos[].url 에 가공본 URL만 넣음
```

- **원본/비공개 앨범의 서명 URL을 페이로드에 그대로 넣지 않는다** — 발행 후 만료로 깨지거나, 영구 공개 시 프라이버시 붕괴(§7).
- **EXIF/GPS 제거를 검증**한다 (P5 DoD: "발행 사진 EXIF 제거 검증"). 가공본에 GPS·원본 촬영기기 정보가 남으면 테스트 실패.
- 발행 직전 **공개 동의 다이얼로그**를 띄운다("이 사진들이 공개 블로그에 올라갑니다").
- GitHub 토큰 등 운반책 키도 프록시 경유. 자동 발행 대신 `_drafts`/PR로 사람이 확인(§7 운반책 2).

## 4. 삭제 = 휴지통 (soft-delete) (§4.3)

- 삭제는 `deleted_at`만 채운다. "상대가 지운 우리 추억"을 양쪽 모두 **복구 유예 기간** 동안 복구 가능(§8 출처·정서 철학과 일관).
- 기본 쿼리/RLS는 `deleted_at IS NULL`만 노출. 휴지통 화면에서만 복구/영구삭제.
- 물리 삭제는 (a) 복구 유예 경과 후 정리 잡, (b) 관계 종료 시 삭제 정책(§5), (c) 법적 삭제 요청에서만.
- 낙관적 락: 삭제/복구도 `version` 조건부 update로 충돌 감지(LWW 금지, §4.3).

## 5. 관계 종료 시 데이터 소유권 (§10.4) — 1급 요구사항

커플 앱의 현실(헤어짐·계정 분리·삭제 요구)을 설계에 포함한다.

### 5.1 연결 해제(DISCONNECT) 흐름
- `couples.status`를 `ACTIVE → DISCONNECTED`로. 이후 양쪽의 신규 공유 쓰기를 차단(RLS/앱 레이어).
- 연결 해제 시 **공유 데이터 사본 정책을 명시**: 누가 무엇을 보관/삭제하는지를 UI에서 분명히 고지하고 선택하게 한다(예: 양측 각자 내보낸 뒤 공유본 삭제, 또는 한쪽 보관).

### 5.2 양측 동등 내보내기 (권리 — 양쪽 모두에게 동등)
- **사진 ZIP + JSON** 덤프를 양쪽 모두 받을 수 있어야 한다. 한쪽(소유자)에게만 유리하면 안 됨 — 분쟁·법적 리스크 감소가 목적(§10.4).
- 내보내기는 P0의 **내보내기 v0(JSON 덤프)**에서 시작해 §9.2 백업 트랙으로 점진 강화(관계 종료 ZIP+JSON 양측 동등까지).
- 사진은 원본 포함(이건 둘의 사적 사본이므로 EXIF 스트립 안 함 — §3.3의 공개 발행과 구분).

### 5.3 삭제 요청 처리
- 일방/양방 **삭제 요청**을 지원(PIPA 삭제권). soft-delete를 넘어 물리 삭제까지 처리하는 경로를 둔다.
- 삭제 전 내보내기 기회를 제공(데이터 회수 권리 보장).

## 6. 법규 요약 (PIPA · 위치정보보호법)

2인용이라도 위치·사진 처리·외부 발행은 국내 법규 원칙을 따른다:
- **동의**: 위치 공유(§3.2), 사진 공유(§3.2), 블로그 공개(§3.3)에 명시 동의.
- **최소 수집**: 기능에 필요한 데이터만 수집·보관. 불필요한 위치/메타데이터 장기 보관 금지.
- **회수·삭제권**: 내보내기(§5.2)와 삭제 요청(§5.3)으로 보장.
- **목적 외 사용 금지**: 둘만의 도구 목적 외로 데이터를 쓰지 않음(제3자 공유·광고 없음).

| P1 | RLS 격리 테스트 CI 게이트, soft-delete + 낙관적 락(§4), 자동완성 프록시 키 격리(§1) |
