# PROMPT — love_place 매뉴얼-모드 코딩 kit 실행 드라이버

> 이 파일은 **사용자가 로컬 Claude Code 세션으로 따르는 지시서**입니다.
> 패킷 단위로 코딩할 때 매번 이 문서를 기준으로 다음 행동을 정하세요.
> 모든 세부는 다른 kit 파일에 있습니다 — 여기서는 **경로로 가리키기만** 합니다(복붙 금지).

---

## 1. 이 앱이 뭔지 (한 문단)

**love_place**는 나와 여자친구 **둘만 쓰는 여행 관리 웹앱**입니다(2인 전용, 회원가입 퍼널·권한 등급 없음). 가고싶은/가본 장소를 카카오 자동완성으로 모아 지도에 별표 마커로 보고, 3트랙 공유 캘린더로 일정을 맞추고, 여행(Trip)별·지역별로 사진 앨범을 쌓고, 지역 데이터가 모이면 AI가 일자별 코스를 짜 지도·캘린더로 흘려보냅니다. **공유가 기본값**이고 개인 영역(내 일정)만 분리되며, "누가 추가했는지" 출처 표시와 둘 사이 실시간 동기화가 핵심 경험입니다. 플랫폼은 확정된 **React + Vite + TypeScript PWA + Supabase**(설계서 §2 B안을 '웹앱'으로 해소)이며, 모든 외부 API 키는 Supabase Edge Function 프록시 뒤에 숨깁니다.

---

## 2. 시작 전 준비 (코딩 첫 줄 전에 반드시 완료)

스택·키·외부 프로젝트가 준비되지 않으면 P0 후반부와 P1이 공회전합니다. 코딩 착수 전 **스택 결정 패킷의 체크리스트**를 끝내세요.

- 따라야 할 파일: **`.ai-factory/00-stack-decision.md`** (키·외부 프로젝트 준비 체크리스트의 정본)
- 거기서 확인·발급하는 항목(요약, 정본은 그 파일):
  - **Supabase 프로젝트** 생성 → `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` 확보, Auth(매직링크/OAuth) 활성화.
  - **Edge Function 시크릿**(클라이언트에 절대 노출 금지): 카카오 REST 키, 카카오모빌리티/TMap 키, Anthropic API 키 → `supabase secrets set` 으로만 보관.
  - **카카오맵 JS SDK 앱 키**(지도 표시용, 웹 도메인 등록) — 표시용 JS 키와 프록시용 REST 키를 혼동하지 말 것.
  - 외부 무료 한도 재확인(설계서 §5.5·§6·§9.2 TCO): 카카오맵 SDK 한도, 카카오 로컬/모빌리티·TMap 일 한도, Anthropic 단가.
  - `.env.local`은 `VITE_*` 공개 키만. **API 키(카카오 REST·Anthropic 등)는 `.env`에도 두지 않음** — 전부 Edge Function 시크릿.
- 게이트: 위 체크리스트가 전부 ✅ 되기 전에는 P0의 Supabase/프록시 태스크를 시작하지 마세요.

## 3. 읽는 순서 (코딩 전 컨텍스트 적재)

매 세션 시작 시 아래를 이 순서로 읽어 컨텍스트를 맞춥니다.

1. **소스 오브 트루스** — `여행관리앱_설계서.md` (해당 패킷이 인용하는 §절 위주로). 모든 결정의 최종 근거.
2. **`.ai-factory/`** (kit 스파인) — 이 디렉터리의 파일들을 순서대로:
   - `.ai-factory/00-stack-decision.md` — 확정 스택·키 준비(§2 참조).
   - `.ai-factory/01-spec.md` — 기능 스펙/AC(설계서 §5 증류).
   - `.ai-factory/02-data-model.md` — 테이블·컬럼·RLS·동기화/도출 규칙(설계서 §4 증류).
   - `.ai-factory/03-proxy-contract.md` — Edge Function 프록시 입출력 계약(§2.1·§5.6).
   - `.ai-factory/04-tasks.md` — 패킷 계획(P0~P5)과 패킷별 태스크·DoD. **실행 루프의 진입점**.
3. **`.claude/rules/`** (코딩 규약, 모든 패킷에 강제) — 이 디렉터리의 규칙 파일 전부. TS strict, API 키 클라이언트 금지(전부 Edge Function 프록시), RLS 전 테이블, soft-delete + `version` 낙관적 락 + 오프라인 큐, 색만으로 상태 구분 금지(색+패턴/라벨), 접근성(VoiceOver/Dynamic Type/Reduce Motion/다크모드) 기본, 빈 상태/로딩 디테일 필수.

> 위 디렉터리에 아직 파일이 없으면, 해당 패킷이 그 파일을 **생성**하는 것이 첫 태스크입니다(`.ai-factory/04-tasks.md`의 P0 참조). kit 파일은 설계서를 '구현 계약'으로 증류한 것이지 복제가 아닙니다 — 스파인과 모순되면 설계서를 신뢰하세요.

## 4. 패킷 실행 루프

`.ai-factory/04-tasks.md`의 **P0부터 순서대로**. 한 번에 한 패킷, 패킷 안에서 한 태스크씩:

1. **패킷 읽기** — `.ai-factory/04-tasks.md`에서 현재 패킷(예: P0)과 그 태스크 목록·DoD·인용 §절을 읽는다. 해당 §절을 설계서에서 확인한다.
2. **TDD/구현** — 태스크별로:
   - 테스트 먼저(동기화/충돌/오프라인은 **P1부터 필수**, 설계서 §9.2 횡단 트랙). `src/__tests__/`에 vitest.
   - 지정 절대경로에 구현(폴더 구조: `src/{pages,components,lib,hooks,state,styles,__tests__}`, `supabase/{migrations,functions/<edge-fn>}`, `e2e/`).
   - 규약 위반 금지(§3.3의 `.claude/rules/*`): 클라이언트 키 금지, RLS, soft-delete+version, 색+라벨, 접근성, 빈/로딩 상태.
3. **게이트 통과** — §5의 게이트를 **전부** 돌려 초록일 때만 다음으로.
4. **다음** — 패킷 내 다음 태스크. 패킷의 모든 태스크 + 패킷 DoD가 충족되면 다음 패킷으로. **건너뛰지 말 것.**

각 변경은 설계서 **§절 번호를 근거로 인용**해 커밋 메시지/코멘트에 남깁니다(예: "위시 중복 = kakao_place_id 점프, 설계서 §5.2").

## 5. 패킷별 게이트 (전부 초록이어야 통과)

매 태스크/패킷 종료 시 아래를 실행. 하나라도 빨가면 다음으로 넘어가지 않습니다.

| 게이트 | 명령(예) | 합격선 |
|---|---|---|
| **타입체크** | `npx tsc --noEmit` | 에러 **0** (TS strict) |
| **단위 테스트** | `npx vitest run` | 전부 통과. 동기화/충돌/오프라인 케이스 포함(P1부터) |
| **빌드** | `npm run build` | 성공 |
| **비주얼 스모크** | `npx playwright test`(e2e/) | 핵심 화면 스모크 통과 |
| **RLS 격리** | `.ai-factory/04-tasks.md` P0의 RLS 격리 테스트 | **다른 couple_id 데이터에 접근 0건**, visibility(개인/공유) 다단계 정책 확인 |
| **프록시 보안** | P0 Edge Function 테스트 | 인증 없는 호출 거부, 레이트리밋·월 상한 동작(설계서 §2.1) |

> 게이트는 `.claude/rules/*`와 `.ai-factory/04-tasks.md`의 패킷별 DoD가 정본입니다. 위 표는 공통 최소선입니다.

## 6. 실행 순서 — P0 먼저, 그 다음 P1 MVP 핵심 루프

**반드시 이 순서.** 데이터가 쌓여야 가치가 생기는 앱이므로 토대 → 핵심 루프 순으로 갑니다(설계서 §9.1).

- **먼저 P0 (토대)** — Vite+React+TS 스캐폴드, Supabase 프로젝트·Auth, `couples`/`profiles` + 초대·연결(1회용·만료 코드, 1:1 바인딩), RLS 골격, Edge Function 프록시 스켈레톤(인증·레이트리밋·상한), 내보내기 v0(JSON 덤프), CI.
  **P0 DoD:** 둘이 연결되고 RLS로 couple 격리 확인 + 프록시 인증·상한 동작. **이게 초록이기 전엔 P1을 시작하지 않습니다.**
- **그 다음 P1 (MVP 핵심 루프)** — 위시 장소(카카오 자동완성 디바운스 250ms→프록시, `places`+`wishes`, 중복=`kakao_place_id` 점프, stale 취소토큰) + 지도 별표 마커(가고싶음=빈별/가봤음=채운별+체크, 클러스터링, 출처 아바타) + 공유(Realtime) + 충돌/오프라인 기본.
  **P1 DoD:** 장소 저장 ≤3탭, 자동완성 체감 ≤400ms, 약전파 유실 0.
- 이후 **P2 일정 → P3 기록 → P4 차별화(추천·AI 경로) → P5 마감(blog auto writer·폴리시)** 순으로, 각 패킷의 DoD를 게이트로 잠그며 진행(설계서 §9.1, 상세는 `.ai-factory/04-tasks.md`).

> 푸시 인프라는 끝으로 미루지 말고 알림이 필요한 패킷(P2 일정 리마인더 등)으로 당겨 구축하되, iOS 웹푸시 제약 때문에 **인앱 활동 피드를 1차 알림 수단**으로 둡니다(설계서 §8).
